단일 RDB 병목 분석 및 1차 성능 최적화
문제

[단일 RDB 병목 지점 확인]

MySQL만 사용하던 때 좋아요 여부를 저장하는 테이블과 게시글의 like_count를 분리해서 관리. 당시에는 좋아요 저장과 카운트 갱신을 서로 독립된 작업으로 생각해 REQUIRES_NEW 전파 속성을 사용. 하지만 이것 때문에 하나의 요청이 동시에 두 개의 커넥션을 점유하는 쓰기 병목 발생. 또, 좋아요/조회수 이벤트가 발생할 때마다 MySQL UPDATE가 실행되어 쓰기 부하가 증가, 3.19 TPS에서도 한계에 도달.


[인기글 기능 도입]

인기글 목록 기능은 실시간 지표를 기반으로 지속적인 정렬 및 조회가 필요해 읽기/쓰기 부하가 동시에 증가, 30 RPS 수준의 부하에서도 p(95) 14초 이상 걸리는 문제 발생.

해결
  • REQUIRED 단일 트랜잭션 구조로 재설계
  • 카운트 증가 이벤트를 메모리에서 우선 집계
  • 스케줄러로 일정 주기마다 DB에 일괄 반영

[인기글 인덱스 최적화]

EXPLAIN을 통해 인기글 조회 쿼리에 풀스캔이 발생한다는 것을 확인, 인기글 점수 계산의 로직에 맞춰 인덱스(created_at)를 추가해 최근 7일 게시물로 대상을 제한해 스캔 범위를 축소.

문제

ASG 인프라 환경에서의 정합성 문제 발생

게시글의 메타데이터를 인메모리에 저장하기 때문에 웹 서버를 다중 인스턴스로 확장하면 각 인스턴스간 상태 불일치 문제 발생. 인스턴스 축소 시, 서버 로컬 메모리에 머물러 있던 데이터가 유실되는 결함 확인.

해결

Redis 기반 Shared State 아키텍처 도입

  • 모든 WAS가 동일한 Counter를 사용하는 Shared State 구조로 전환
  • 좋아요/조회수/댓글수 카운터를 Redis로 이관
  • INCR 기반 원자적 증가로 분산 환경에서도 갱신 손실 방지
  • 5분 주기 Bulk Update 파이프라인 구축
  • Redis Sorted Set 기반 인기글 랭킹 구현
성과

Redis를 Shared State로 활용

단순 캐시 서버 추가가 아닌 Stateful한 서버 구조를 Stateless 아키텍처로 전환하고, Auto Scaling 환경에서도 데이터 정합성과 수평 확장성을 동시에 확보.

문제

비용과 운영성을 고려한 NAT

보안을 위해 Private 서브넷에 인스턴스를 위치시켰지만, 초기 서비스 규모에서는 NAT Gateway의 높은 고정 비용이 부담.

해결

사용자 트래픽이 아닌 배포, 이미지 Pull, 패키지 업데이트 등의 관리 목적의 트래픽만 NAT를 통과하는 구조임을 고려하여, EC2 기반 NAT Instance를 구축하고 Bastion 역할을 통합.

성과
NAT 유지 비용
$30 → $5 / 월
80% 절감. 초기 서비스 규모에서는 관리형 서비스의 가용성보다 비용 효율을 우선해 결정.
문제

이미지 업로드 구조

사용자가 이미지 업로드 시 모든 파일이 WAS를 거쳐 처리되어 네트워크, 컴퓨팅 자원이 불필요하게 소모.

해결

Pre-signed URL을 활용해 클라이언트 → S3 직접 업로드하는 구조로 변경하여 서버를 업로드 경로에서 제외.

성과
  • 서버를 거치는 미디어 트래픽 제거
  • WAS의 컴퓨팅 비용 감소
  • 애플리케이션을 업로드 경로에서 제거하여 서비스와 파일 전송을 분리

배포 안정성과 효율을 고려한 CI/CD 파이프라인 설계
문제

무중단 배포

기존 배포 방식은 서비스 업데이트 시 일시적인 서비스 중단이 존재. 새로운 버전이 정상 실행되지 않은 상태에서 트래픽이 전환되면 서비스 장애 발생.

해결

Blue-Green 배포를 적용하고, Health Check 성공 이후에만 Nginx에서 트래픽을 전환하도록 설계, 배포 실패 시에는 기존 환경을 유지하도록 자동 롤백을 구성.

파이프라인: 코드 푸시 → 빌드 & 테스트(Maven/JUnit) → Docker 빌드(Multi-stage) → 이미지 푸시 → CD 트리거 → Blue/Green Health Check → 트래픽 전환 또는 자동 롤백

성과
배포 중 다운타임
0초 달성

Health Check 실패 시 자동 롤백, 배포 실패가 사용자 장애로 이어지지 않는 구조 확보.

문제

Docker 이미지 최적화

Docker 이미지 크기가 커 빌드와 배포 시간이 증가. Root 권한 실행으로 컨테이너 보안 위험이 존재.

해결

Multi-stage Build와 Layered JAR를 적용하여 변경 레이어만 재빌드하도록 개선, 컨테이너를 Non-root User로 실행하도록 변경.
