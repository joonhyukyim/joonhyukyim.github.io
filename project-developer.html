Cloud Migration · Detail 1
무중단 클라우드 마이그레이션 전략
문제

MVP 단계에서는 GCP 무료 크레딧만으로 개발과 기능 검증이 가능했으나, 서비스 출시 이후 실제 사용자 기반으로 전환되면서 지속 가능한 운영 환경이 필요해 AWS로의 마이그레이션 진행하기로 결정.


운영 중인 서비스였기 때문에 마이그레이션 과정에서도 사용자가 계속 이용할 수 있어야 함. 백업 및 복구 방식은 다운타임이 길어 운영 환경에서 부적합. Double Write 방식은 운영 중인 비즈니스 로직을 수정해야 하고 부분 실패까지 고려해야 하는데, 단 한번의 마이그레이션을 위해 진행하기엔 운영 리스크가 크다는 결론을 내림.

해결
  • AWS DMS의 CDC를 활용해 실시간 변경사항을 지속적으로 복제
  • 컷오버 직전 애플리케이션을 ReadOnly로 전환하여 새로운 쓰기 요청을 잠시 차단
  • Replication Lag가 0초임을 확인한 뒤 컷오버를 진행
  • 마이그레이션 전후 레코드 수를 비교하여 데이터 유실이 없음을 확인
성과

데이터 유실 0건, DB 데이터 정합성 유지


서비스 중단 없이 AWS 환경으로 이전 완료


* 새로운 DB, 컷오버 이전/이후 레코드 개수 3724로 일치 확인

Cloud Migration · Detail 2
무중단 트래픽 전환 전략
문제

마이그레이션 테스트 중 Route53 TTL을 1초로 설정하면 트래픽이 즉시 신규 환경으로 전환될 것이라 생각했지만, 일부 사용자는 기존 서버에 계속 접근하는 문제 발생.


검증: k6 부하 테스트 결과 최대 70초 동안 접속 실패가 발생. DNS TTL을 단축해도 DNS Cache와 기존 TCP 연결 유지로 인해 모든 사용자의 트래픽이 즉시 전환되지 않음.

해결
  • 기존 GCP 서버를 즉시 종료하지 않고, DNS가 갱신되지 않은 사용자의 요청을 Nginx Proxy Forwarding을 통해 AWS 서버로 전달
  • 사람이 직접 여러 단계를 수행하면서 발생할 수 있는 실수와 지연을 줄이기 위해 Shell 스크립트로 컷오버 과정을 자동화
  • Source DB ReadOnly 전환 → 최종 동기화 확인 → 트래픽 전환을 단일 스크립트로 실행

DoS Defense · 1st Response
대규모 DoS 공격 분석 및 1차 대응
문제

서비스 운영 중 공개 API를 대상으로 초당 약 270건 규모의 DoS 공격으로 서비스 중단. 로그 분석을 통해 단일 IP에서 대량 요청이 발생하며 Nginx Connection이 고갈했고 CPU 사용률이 100%에 도달하며 애플리케이션까지 요청이 도달하지 못해 서비스가 정상 요청을 처리하지 못한 것을 확인. 추가로 장애 인지에 30분 이상이 걸려 복구 및 대응이 늦어지는 문제 발생.

조치
  • Nginx Connection을 기존 768에서 2048로 증설
  • API 중요도별 Rate Limiting 적용
  • Prometheus + Loki + Grafana(PLG 스택)로 모니터링 시스템 도입
  • Alert Manager와 Discord를 연동해 장애 발생 시 실시간 알림 발송
# /etc/nginx/nginx.conf limit_req_zone $binary_remote_addr zone=login_limit:10m rate=10r/s; limit_req_zone $binary_remote_addr zone=api_limit:10m rate=100r/s; limit_conn_zone $binary_remote_addr zone=addr:10m;

단일 IP당 초당 최대 100개 요청으로 제한, 로그인 API는 초당 10개로 강력 제한

성과 · 한계

150개 요청 테스트를 통해 Rate Limiting의 정상 작동 확인


에러 탐지부터 로그 분석까지의 MTTR을 5분 이내로 단축


한계: 이 같은 방법으로 트래픽 폭주를 방어하려고 했으나 규모가 더 커진 2차 공격에서 애플리케이션 레벨 방어의 한계를 확인 (다음 슬라이드)

DoS Defense · 2nd Response
대규모 DoS 공격 분석 및 2차 대응
문제

동일한 API로 3,300 req/s 수준의 2차 대규모 DoS 공격 발생. Rate Limiting은 정상적으로 동작했지만, 수백만 건의 503 에러를 생성하고 TCP 연결을 맺고 끊는 과정에서 인스턴스의 CPU 사용률이 100%까지 상승.


2차 공격 당시: CPU 사용률 10-20% → 100%, Active Connections 100-300 → 2,048 (완전 고갈), Waiting Connections 0-50 → 2,048 (대기 큐 포화), Error Log 999,726건 (5분간)

해결

iptables를 활용해 Nginx 도달 전에 비정상 IP로부터 오는 패킷을 즉시 드랍하도록 Fail2Ban 자동 차단 시스템을 구축. 인증이 필요 없는 공용 API에 대해 요청 제한을 더욱 강화하여 신규 공격 유입 자체를 감소.

maxretry = 5 findtime = 60 bantime = 600 action = iptables-multiport[name=ReqLimit, port="http,https", protocol=tcp] discord-notify

1분 내 Rate Limit 5회 초과 시 iptables로 커널 레벨에서 ip 차단

문제

Kubernetes & Network
Kubernetes 운영 및 네트워크 최적화
문제
  • Docker + ASG 환경에서 사용하던 ALB를 Kubernetes에 그대로 적용하자, AWS ALB와 Ingress Controller가 모두 L7에서 HTTP를 분석하는 이중 프록싱 구조가 되어 불필요한 라우팅 오버헤드가 발생
  • ArgoCD SelfHeal과 HPA가 Deployments의 replicas 필드를 동시에 관리하면서 원하는 상태 복원과 오토스케일링이 반복되며 충돌
해결
  • ALB를 NLB로 변경하고 Gateway API(Envoy)를 도입해 외부는 L4 Pass-through, 내부에서만 L7 라우팅을 수행하도록 변경해 중복 HTTP 파싱을 제거
  • Deployments의 replicas 선언을 제거하고 ArgoCD의 ignoreDifferences 정책을 적용하여 HPA가 replicas를 전담하도록 변경. GitOps는 선언적 상태를, HPA는 동적인 스케일링만 담당하도록 역할을 분리
